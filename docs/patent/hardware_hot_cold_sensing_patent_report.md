## 一、背景

### 1.1 異質記憶體架構的興起

現代資料處理系統——包含 AI 加速器、高效能運算系統、資料中心處理器——普遍採用異質記憶體架構。系統中同時存在多種記憶體層級，各層級在存取延遲、頻寬、容量等特性上差異顯著：

- **近端記憶體（near memory）**：與運算單元同封裝，延遲低、頻寬高，但容量有限。典型代表為 HBM（High Bandwidth Memory），以 3D 堆疊方式與處理器晶片共同封裝，提供數 TB/s 等級的頻寬。
- **遠端記憶體（far memory）**：經由晶片間互連或擴展介面存取，延遲較高，但容量大幅提升。包括 Host DDR DRAM（透過傳統記憶體通道連接）、CXL-attached DRAM（透過 CXL 協定擴展的外接記憶體），以及 LPDDR（經由 interposer 或 bridge 連接）。

此外，在多運算單元的 SoC 中，不同運算單元到不同記憶體的拓撲距離不等（NUMA 特性），使得「同一筆資料放在不同記憶體位置」對效能的影響更加不均。

### 1.2 Cache Mode 與 Flat Mode

在異質記憶體架構中，近端記憶體與遠端記憶體之間的關係存在兩種根本不同的組織方式：**cache mode** 與 **flat mode**。理解這兩種模式的差異，是理解本案技術貢獻的前提。

**Cache mode** 是指將近端記憶體（如 HBM）當作遠端記憶體（如 DDR）的最後一級快取（last-level cache）。在此模式下，HBM 不具備獨立的位址空間，軟體看到的只有 DDR 的位址。當處理器存取資料時，硬體自動檢查資料是否已快取在 HBM 中——若命中則直接回傳，若未命中則從 DDR 載入至 HBM，並在 HBM 空間不足時以固定的替換策略（如 LRU）自動驅逐舊資料。整個過程對軟體完全透明，無需任何資料放置決策。早期的異質記憶體系統（例如 Intel Knights Landing 處理器的 MCDRAM cache mode）即採用此設計。

**Flat mode** 則是將近端記憶體與遠端記憶體視為獨立的、各自可定址的記憶體空間。HBM 與 DDR 各自擁有獨立的位址範圍，軟體可以明確指定每一筆資料應存放在哪個記憶體層級。這意味著資料放置的決策權從硬體轉移到了軟體——硬體不再自動搬遷資料，而是由軟體（或輔助硬體機制）根據資料的存取行為主動決定放置位置。

當前業界主流的 AI 加速器——如 NVIDIA H100、AMD MI300X——均採用 flat mode。CXL-attached memory 從協定本質上即為 flat mode，因為 CXL 記憶體裝置在系統中呈現為獨立的位址區間。可以說，**flat mode 已成為高效能異質記憶體架構的主流選擇**。

業界從 cache mode 轉向 flat mode，有以下幾個關鍵原因：

1. **容量損耗**：Cache mode 需要為每個快取行（cacheline）維護 tag 元資料（用於記錄該行對應的 DDR 位址）。當 HBM 容量達到數十 GB 時，tag 所佔的儲存空間已不可忽略，相當於浪費了寶貴的近端記憶體容量。
2. **Cacheline 粒度引發頻繁顛簸（thrashing）**：Cache mode 以 cacheline（通常 64 bytes）為搬遷單位。當工作集（working set）略微超過 HBM 容量時，資料會在 HBM 與 DDR 之間以極高頻率來回搬遷，造成嚴重的效能下降。AI workload 中常見的情境——如 MoE 模型的 expert 切換——恰好容易觸發此問題。
3. **缺乏預測能力**：Cache mode 的替換策略僅依據過去的存取歷史（如 LRU）做被動決策，無法預測哪些資料即將被存取。對於具有明顯階段性存取模式的 AI workload（如依序處理模型各層的 weights），這種被動策略無法提前備妥所需資料。
4. **粒度不匹配**：AI workload 的資料搬遷需求通常以 tensor 或 expert 為單位（數十 KB 至數百 MB），而非 cacheline 的 64 bytes。Cache mode 的細粒度搬遷在此場景下既低效又缺乏語義。

### 1.3 資料溫度的本質

在 flat mode 架構中，資料放置決策由軟體負責，因此系統需要一種機制來判斷各記憶體區域的存取活躍程度。**並非所有資料在任何時刻都同等重要**。以 AI workload 為例：

- 正在被運算單元使用的 layer weights，需要最低延遲存取
- 數秒前已計算完畢的 layer weights，短期內不再被使用
- MoE 模型中當前 token 啟用的 expert，正在被密集存取
- MoE 模型中未被啟用的 expert，閒置中

我們以**溫度（temperature）**來描述資料的存取活躍程度。正在被頻繁存取的資料具有較高的**溫度分數（temperature score）**，稱為熱資料；不常被存取的資料溫度分數較低，稱為冷資料。當某筆資料的存取模式發生顯著改變——例如從冷變熱——即構成一個**溫度轉換事件（temperature transition event）**。理想情況下，熱資料應放置在近端記憶體（HBM）以獲得最低延遲，冷資料可放置在遠端記憶體（DDR 或 CXL memory）以釋放近端容量。

### 1.4 Flat Mode 下的管理空白與本案邊界

在晶片內部的 cache hierarchy 中（L1、L2、L3 等），硬體以固定的替換策略（如 LRU）自主管理資料的留存與驅逐。雖然部分架構提供有限的軟體提示機制（如 ARM 的 PLD prefetch 指令、x86 的 PREFETCHNTA），但 cache hierarchy 的核心定位仍是對軟體透明的自動管理，並不提供跨記憶體層級的主動感知與預測能力。

在 cache hierarchy 以外、flat mode 下的多層級 DRAM 記憶體之間（HBM、Host DDR、CXL memory），目前缺乏對等的硬體管理機制。這些記憶體層級之間的資料放置決策，仍仰賴作業系統軟體以毫秒級的輪詢方式執行，或由程式設計師靜態指定，不具備主動感知與預測能力。

**本案的涵蓋範圍**：本案聚焦於 flat mode 下的 DRAM 等級記憶體層級放置問題，涵蓋 HBM、Host DDR DRAM、CXL-attached DRAM 之間的資料搬遷決策。本案不涉及 cache hierarchy 內部的管理（L1/L2/L3 快取、on-chip SRAM、scratchpad memory），也不涉及 HBM 以 cache mode 運作時的資料管理——因為 cache mode 下的資料放置已由硬體快取機制自動處理，不需要額外的溫度感知。

這個 flat mode 下的管理空白，正是本案要填補的。

## 二、要解決什麼問題

### 2.1 問題的精確描述

在具有異質記憶體架構的資料處理系統中，運算所需的資料總量往往超過近端記憶體的容量。以 AI 加速器為例，大型語言模型的 weights 可達數百 GB，而單一加速器的 HBM 容量通常在 80 至 192 GB 之間，不足以容納全部資料。因此，在 flat mode（HBM 與 DDR 為獨立可定址記憶體空間，軟體決定資料放置）的系統組態下，資料需分佈在不同 DRAM-class 記憶體層級中——近端（HBM）、遠端（Host DDR）、擴展（CXL memory）——並在運算過程中動態搬遷：將即將被使用的資料搬入近端記憶體，將不再被使用的資料搬出至遠端記憶體。

此搬遷決策的核心挑戰在於：**如何即時判定各記憶體區域的溫度狀態，並在適當時機觸發搬遷。** 放置決策的品質直接影響效能——高溫資料放在近端記憶體可獲得低延遲存取，若誤留在遠端記憶體則需承受數倍的存取延遲。

問題的時間尺度差距尤為關鍵。以 Mixture-of-Experts（MoE）推論為例，每個 token 由 gating network 動態選擇少數 expert 執行運算，expert 的存取模式在毫秒內即可發生劇變。研究顯示，在 MoE 推論中 expert loading 佔推論時間的 85%–95%（HOBBIT, 2024），這意味著 expert 搬遷效率幾乎直接決定端到端推論速度。然而，現有搬遷決策機制的感知週期在秒級甚至十秒級，與 AI workload 毫秒級的存取模式變化之間存在數個數量級的落差。

### 2.2 現有搬遷決策機制及其限制

以下彙整六類現有機制，涵蓋從純軟體到硬體輔助的各種做法，逐一分析其運作方式與限制：

| 類別 | 代表技術 | 做法 | 限制 |
|------|---------|------|------|
| 軟體靜態配置 | numactl、membind API | 程式設計師或部署工具在啟動時指定資料的記憶體放置位置，運行期間不變 | 純靜態，無法適應運行期間動態變化的存取模式；需要程式設計師對 workload 有完整先驗知識 |
| Fault 驅動遷移 | CUDA Unified Memory | GPU MMU 偵測到 page 不在本端記憶體時觸發 page fault，由 driver 將 page 從遠端搬入近端 | 完全被動——必須等到實際存取 miss 發生才觸發搬遷，搬遷期間運算單元 stall；無預測能力，每次首次存取都付出 fault 代價 |
| 作業系統掃描 | Linux AutoNUMA、TPP（Transparent Page Placement） | 定期將部分頁面的 PTE access bit 清除或標記為 not-present，藉由後續存取觸發的 page fault 或下次掃描時的 access bit 狀態判斷冷熱，統計後決定搬遷 | 掃描週期典型為 10–60 秒，對 AI workload 毫秒級的存取模式變化而言反應太遲；掃描本身消耗 CPU 資源，且頁面標記清除引發額外 TLB shootdown 開銷 |
| HBM cache mode | 部分處理器架構之 HBM cache 組態 | HBM 作為 DDR 的 last-level cache，以 cacheline 粒度（64B）由硬體自動管理資料留存 | cacheline 粒度（64B）導致 tag 儲存開銷佔 HBM 容量的 3%–12%，浪費寶貴的高頻寬記憶體；當 working set 略微超過有效 cache 容量時易發生 thrashing；硬體替換策略固定（如 LRU），無預測能力，軟體無法調整判定標準 |
| GPU access counter | NVIDIA Volta 及後續架構之 access counter（US10,361,722） | 在 GPU 端為每個 page 維護硬體存取計數器，當計數超過可配置閾值時向 driver 發送通知，driver 決定是否遷移 page | 僅計數單一端點的存取，無跨裝置全域視野；基於累積計數的閾值觸發，無法區分存取密度的時間變化；不具備預測能力——只能回報已發生的存取，無法預判即將被存取的區域 |
| CXL 3.1 CHMU | CXL 3.1 規範定義之 Component Hot-tracking and Monitoring Unit | CXL memory device 內部追蹤各 region 的存取熱度，維護 hotlist ring buffer，以 interrupt 或 polling 方式通知 host；內部追蹤演算法為 implementation-defined | 僅限裝置端視角，無法觀察其他記憶體層級的存取狀況，缺乏跨裝置全域視野；內部演算法為各實作自行定義，host 無法調整或統一管理；不具備預測能力——僅回報當前已觀察到的熱區域 |

**六類機制的共同根本限制：** 上述所有機制——無論是軟體掃描、fault 驅動、硬體計數器、或裝置端熱度追蹤——皆只能回報**當前已被存取**的記憶體區域。沒有任何一種機制能夠在資料**尚未被存取之前**，預判其即將變熱並提前通知系統準備搬遷。這是現有技術的根本性缺口：從「感知」到「決策」之間缺乏預測維度，所有搬遷都是事後反應，而非事前準備。

### 2.3 適用場景與不適用場景

本案所針對的問題並非適用於所有記憶體管理情境。以下明確界定本案具優勢的場景，以及現有技術已足夠處理、不需要本案的場景。

**適用場景：存取模式在運行期間動態變化，且存在可學習的統計規律**

| 場景 | 為何適用 | 適用強度 |
|------|---------|---------|
| MoE expert routing | 每個 token 由 gating network 動態選擇少數 expert，被選中的 expert 在各 token 間不斷切換；gating 決策不具跨 token 記憶，無法由前一 token 的選擇直接推導下一 token 的選擇。然而 expert 的啟用在統計上存在可觀察的規律——特定 expert 組合傾向共同出現，且啟用頻率隨輸入分佈呈現可學習的模式 | 高 |
| 推薦系統 embedding table | 每次請求動態查詢不同的 embedding entry，熱門 entry 集合隨趨勢話題、時段漂移；不同 entry 之間存在共現模式（co-occurrence pattern），可從歷史存取序列中學習 | 高 |
| 多模型服務（multi-model serving） | 流量動態決定載入哪個模型，各模型的 weights 需在記憶體層級間搬遷；模型載入需求隨時段（time-of-day）呈現可學習的週期性模式 | 中 |

上述場景的共同特徵：存取模式在運行期間以毫秒至秒級變化，無法在 compile-time 預知，但變化並非完全隨機——存在可從歷史溫度轉換事件（temperature transition event）序列中學習的統計規律。

**不適用場景：存取模式可在 compile-time 預知，或變化速度在現有機制的感知範圍內**

- **LLM layer-by-layer 推論**：Transformer 模型的各 layer 按固定順序依次執行，存取順序在 compile-time 即可完全確定。軟體排程器（如 FlexGen）可預先規劃每一層的資料搬遷時機，不需要運行期間的動態感知與預測。
- **KV cache 管理**：KV cache 隨 sequence length 線性增長，增長模式完全可預測，不存在需要學習的動態模式。記憶體配置可由軟體依 sequence 長度預先規劃。
- **通用伺服器 NUMA 記憶體管理**：一般伺服器 workload 的記憶體存取模式變化速度在秒級，作業系統毫秒級的掃描機制（如 AutoNUMA）已足以應對，不需要更精細的硬體感知與預測。

### 2.4 預計解決方向

本案提出一種資料溫度感知方法，適用於 flat mode 下 DRAM-class 記憶體層級之間的搬遷決策輔助。此方法以多維度存取行為評估產生各記憶體區域之溫度分數（temperature score），並從溫度轉換事件（temperature transition event）的序列中自主學習規律以預測後續轉換，產生放置建議（placement hint）。其目標在於填補現有技術的根本性缺口——從僅能回報「已發生之存取」的被動感知，進化為能夠預判「即將發生之存取」的主動預測，使搬遷決策得以在資料被實際存取之前提前準備。

## 三、解決方法

在具有異質記憶體架構的資料處理系統中，各記憶體區域的存取活躍度隨運算階段持續變化。部分區域正被密集存取，部分區域已不再被使用但仍佔據近端記憶體空間，另有部分區域即將被存取但尚未被搬入。如何即時且準確地判定各區域的溫度狀態，並在資料**尚未被存取之前**預判其即將變熱、提前觸發搬遷準備，直接決定系統效能。

習知做法中，溫度判定依賴作業系統以毫秒級輪詢掃描，或由記憶體裝置端獨立計數後回報——前者反應遲緩，後者僅看到各自端的局部流量，兩者皆只能回報當前已被存取的區域，不具備預測未來溫度變化的能力。

有鑑於此，本案提出一種**軟體演算法**，透過多維度存取行為評估與溫度狀態轉換序列學習，自主評估各記憶體區域的溫度狀態並預測即將發生的溫度變化。以下先概述核心演算法，再逐步展開各機制的細節。

### 3.1 核心概念

本案的核心是一種**溫度狀態轉換序列學習演算法（temperature state transition sequence learning algorithm）**——一種運行於系統軟體或韌體層的評估方法。此演算法接收記憶體存取活動資訊（可透過硬體效能計數器、記憶體控制器統計、或作業系統層級的存取追蹤取得），從三個維度評估記憶體區域的溫度：時間維度追蹤存取累積與衰減，空間維度捕捉相鄰區域的共同活躍傾向，序列維度則從歷史溫度轉換事件中學習規律並預測下一個即將變熱的區域。三個維度的評估結果合成為單一溫度分數（temperature score），當分數跨越閾值時產生放置建議。

**與現有軟體方案（如 AutoNUMA、TPP）的根本差異**在於：現有軟體方案只能計數或掃描「已被存取」的區域，本案的演算法能夠從溫度轉換事件的歷史序列中學習規律，預測「尚未被存取但即將變熱」的區域。這是一個純演算法層面的創新——不需要新的硬體單元，只需要從現有硬體基礎設施（performance counter、MC 統計等）取得存取資訊即可運作。

在一些實施例中，此演算法亦可以硬體邏輯實現以獲得更低延遲（§3.8 討論硬體實作考量），但演算法本身不依賴特定硬體。

### 3.2 資料溫度評估引擎

本案包含一個**資料溫度評估引擎（data temperature evaluation engine）**，負責接收記憶體存取資訊、維護各區域的溫度分數、偵測溫度轉換事件、並產生放置建議。評估引擎內部維護一張區域狀態表（region state table），為每個被追蹤的記憶體區域記錄其溫度分數與狀態旗標。

評估引擎從三個維度對區域狀態表中的溫度分數進行更新，三個維度的加分結果共同寫入同一張區域狀態表，合成為單一溫度分數。

**時間維度（temporal dimension）**：當評估引擎收到一筆記憶體存取資訊時，對該存取所對應之區域的溫度分數進行累加。與此同時，評估引擎以可配置的週期對區域狀態表中所有區域的溫度分數執行週期性右移（periodic right-shift），等效於將分數減半。持續被存取的區域其溫度分數維持在高位，未被持續存取的區域其分數因衰減而逐步歸零。時間維度構成溫度評估的基礎，持續追蹤各區域的存取活躍度。

**空間維度（spatial dimension）**：當某個區域發生溫度轉換事件——例如其溫度分數從低於閾值上升至超過閾值——評估引擎對該區域在位址空間中相鄰的區域施加額外加分。此維度的直覺在於：在許多 workload 中，相鄰的記憶體區域往往具有相關的存取模式。空間維度僅在溫度轉換事件發生時觸發，而非每次存取都觸發，因此處理開銷極低。**空間維度的加分強度可配置，包括配置為零以停用。** 這一點在 MoE workload 中尤為重要——MoE 的 expert 在記憶體中的排列順序與其被 gating network 啟用的順序無關，空間相鄰性不提供有用資訊，此時應將空間維度停用。

**序列維度（sequential dimension）**：當某個區域發生溫度轉換事件時，評估引擎向行為學習表（behavior learning table）查詢該區域是否存在已確認的後繼規律。若存在，則對預測的後繼區域施加溫度分數加分——即使該後繼區域尚未被任何運算單元存取。這是本案與所有現有機制的根本差異：序列維度能夠在資料尚未被存取之前，預判其即將變熱。行為學習表的詳細機制將在下一節展開。

**整體運作流程**：存取資訊進入評估引擎後，時間維度對該區域加分。若此次加分使該區域的溫度分數跨越 promote 閾值，構成溫度轉換事件，此事件同時觸發空間維度與序列維度——空間維度對相鄰區域加分，序列維度查詢行為學習表並對預測的後繼區域加分。評估引擎持續監控分數變動，當任何區域的分數跨越提升或降級閾值時，產生放置建議並通知軟體。

### 3.3 行為學習表

行為學習表是序列維度的核心元件，負責從歷史溫度轉換事件的先後順序中學習可重複的規律，並在新的轉換事件發生時輸出預測。

**表結構**：行為學習表由有限數量的 entry 組成。每個 entry 記錄一組前驅/後繼對應關係，至少包含三個欄位：前驅區域識別碼（predecessor region ID）、後繼區域識別碼（successor region ID）、以及信心度計數器（confidence counter）。一個前驅區域可對應多個後繼 entry（不同的 successor），反映同一區域在不同情境下可能有不同的後繼轉換。In some embodiments, the behavior learning table comprises 512 entries.

**學習機制**：行為學習表的學習由溫度轉換事件觸發，而非由每次記憶體存取觸發——這是一個關鍵的設計選擇。當某個區域發生溫度轉換事件時，評估引擎記錄該區域為「最近轉換區域」。當下一個溫度轉換事件發生時，前一個轉換區域即為前驅（predecessor），當前轉換區域即為後繼（successor），評估引擎在行為學習表中查詢是否已存在此前驅/後繼對應。若已存在，將該 entry 的信心度計數器累加；若不存在，則建立新的 entry。當信心度計數器超過可配置的確認閾值時，該前驅/後繼對應關係被視為已確認的規律。

**預測機制**：當新的溫度轉換事件發生時，評估引擎以該轉換區域作為前驅，在行為學習表中查詢所有匹配的 entry。若存在信心度超過確認閾值的 entry，評估引擎對該 entry 的後繼區域施加溫度分數加分——提前「預熱」該區域。若同一前驅對應多個已確認的後繼，以信心度最高的 entry 優先。

**替換策略**：行為學習表容量有限。當所有 entry 已滿且需要記錄新的對應關係時，以信心度最低的 entry 優先替換。

**衰減與失效**：信心度計數器可透過週期性遞減（類似溫度分數的右移衰減），使長期未被重複觀測到的對應關係信心度逐步降低。當信心度歸零後，該 entry 可被新的對應關係替換，實現自動回收。這確保行為學習表能夠適應 workload 隨時間變化的存取模式。

**儲存需求估算（non-limiting example）**：以追蹤 256 個 MoE expert 為例。In some embodiments, the region state table comprises 256 entries, each entry containing a region identifier, a temperature score, and status flags, totaling approximately 1 KB. The behavior learning table comprises 512 entries, each entry containing a predecessor region identifier, a successor region identifier, and a confidence counter, totaling approximately 3 KB. The total storage requirement is less than 8 KB, which is negligible relative to the area of a memory controller.

### 3.4 通知機制與軟體協作

評估引擎產生的放置建議透過 **hotlist** 傳遞給軟體。Hotlist 為 ring buffer 結構，每個 entry 包含區域識別碼（region ID）與建議方向（promote 或 demote）。軟體可透過兩種方式讀取 hotlist：中斷（interrupt）方式——當 hotlist 中的 entry 數量超過可配置的 watermark 時觸發中斷通知；或輪詢（polling）方式——軟體主動檢查 hotlist 是否有待處理的 entry。

此通知路徑與 CXL 3.1 CHMU 的 hotlist ring buffer + interrupt/polling 機制在結構上相似。**本案的差異不在通知機制本身，而在 hint 的內容——CHMU 的 hotlist 只能包含已被存取的 region，而本案的 hotlist 可包含序列維度預測的、尚未被存取的 region。** 這是因為序列維度在觀察到溫度轉換事件後，可對尚未被任何運算單元存取的後繼區域施加加分，使其溫度分數跨越閾值而進入 hotlist。

**感知與搬遷解耦**：評估引擎負責產生放置建議，但不直接執行搬遷。搬遷的決策權在軟體——軟體讀取 hotlist 後，綜合考量當前記憶體容量壓力、搬遷頻寬預算等因素，決定是否以及何時執行搬遷。搬遷本身透過既有的 DMA 機制完成。

**搬遷抑制**：評估引擎為每個區域提供搬遷進行中的抑制旗標（migration pending flag）。軟體在啟動搬遷時設定此旗標，搬遷完成後清除。抑制期間，評估引擎不對該區域產生新的放置建議，避免對正在搬遷中的區域重複發出建議。

### 3.6 Batch Serving 下的序列交錯

在 batch serving 場景中，多個推論請求同時執行，各請求的 expert 啟用在時間上交錯。這可能導致行為學習表記錄到不具因果關係的虛假關聯（spurious correlation）。

以 MoE 推論為例：Request 1 依序啟用 Expert 1 和 Expert 3，兩者之間存在因果關係（同一請求的 gating 決策）。但在 batch serving 中，Request 2 可能在 Expert 1 和 Expert 3 的轉換事件之間啟用了 Expert 42。行為學習表在此情況下可能記錄 Expert 1 到 Expert 42 的對應關係，而非正確的 Expert 1 到 Expert 3。

此問題由信心度機制自然緩解。真正具有因果關係的轉換模式——如 Expert 1 到 Expert 3——在大量推論請求中會反覆出現，信心度計數器持續累加並超過確認閾值。由交錯造成的虛假關聯——如 Expert 1 到 Expert 42——本質上是隨機的，不會在多次觀測中一致地重複出現，因此信心度無法累積至確認閾值。週期性的信心度衰減進一步確保偶發的虛假 entry 最終被回收。信心度閾值越高，對交錯雜訊的過濾效果越強，但同時需要更多觀測次數才能確認真正的規律——此為可配置的精確度與反應速度之間的權衡。

### 3.7 硬體實作考量

在評估引擎以硬體邏輯實現的實施例中，以下設計考量適用。此節描述一種可能的硬體加速實作方式，作為軟體演算法的可選最佳化路徑，不限制演算法的其他實現形式。

- **區域粒度可配置**：每個被追蹤的區域大小由軟體透過 CSR 配置（例如 64KB、2MB 等）。區域狀態表的容量有限，當被存取的區域數超過表容量時，以替換策略（如信心度最低者優先替換）回收 entry。
- **狀態表分組存取**：三個評估維度與週期性右移共四個寫入來源需存取同一張區域狀態表。狀態表可採用分組（banking）結構——依區域識別碼的低位分組，使不同區域的寫入得以並行，可降低衝突機率。
- **雙閾值遲滯判定**：溫度判定採用 promote 閾值與 demote 閾值分離的遲滯機制。分數從下方跨越 promote 閾值時觸發提升（構成溫度轉換事件），從上方跌破 demote 閾值時才觸發降級。兩閾值之間的中間區維持現狀，避免週期性右移造成分數在閾值附近反覆震盪。

  P ┤─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  promote 閾值
    │         ↑ 跨越：觸發 promote（溫度轉換事件）
    │
    │    （中間區：維持現狀，不觸發任何建議）
    │
    │         ↓ 跌破：觸發 demote
  D ┤─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  demote 閾值
    │

## 四、新穎進步簡述

### 習知 vs 本案對比

本案的核心新穎性在於**演算法**——溫度狀態轉換序列的自主學習與預測——而非部署位置。以下對比聚焦於演算法層面的差異。

**CHMU（CXL 3.1 spec 定義的 Hotness Monitoring Unit）** 是目前最接近的硬體前案。其運作方式為：各 CXL Type 3 memory device 內的 CHMU 以有限數量的 counter 追蹤到達自己端的存取（支援 epoch-based 與 always-on 兩種模式，可配置 DPA region filtering、R/W filtering 與 downsampling），counter 超過閾值的 region 被加入 PCI BAR 中的 hotlist ring buffer，軟體可以 polling 或 interrupt（watermark / overflow）方式讀取，Host OS 讀取並彙整多個 device 的 hotlist 後決定搬遷。

**本案** 的資料溫度評估引擎從三個維度——時間維度、空間維度、序列維度——對區域狀態表中的溫度分數進行更新。當任何區域的分數跨越提升或降級閾值時，構成溫度轉換事件，同時觸發空間維度（對相鄰區域加分）與序列維度（查詢行為學習表，對預測的後繼區域加分）。放置建議透過 hotlist（ring buffer）傳遞給軟體，支援 interrupt（watermark）或 polling 讀取。

**誠實聲明**：通知路徑在結構上相似——均為 ring buffer + interrupt/polling + 軟體讀取決策——差異在 hint 內容。本案的 hint 包含序列維度預測的結果，即尚未被存取但預測即將變熱的區域；CHMU 的 hotlist 僅包含已超過計數閾值的區域。

**關鍵差異**：

| 面向 | CHMU（CXL 3.1） | 本案 |
|------|-----------------|------|
| **預測能力** | 無——僅回報已超過計數閾值的區域，屬於事後回報 | 有——行為學習表從溫度轉換事件序列中學習規律，可預測尚未被存取但即將變熱的區域 |
| **多維度評估** | 單維度計數（存取頻率） | 三維度整合——時間維度（衰減式累計）、空間維度（鄰近擴散）、序列維度（轉換序列預測）寫入同一溫度分數 |
| **可通知尚未被存取的區域** | 否——region 必須先被存取且 counter 超過閾值才會出現在 hotlist | 是——序列維度可在前驅區域發生溫度轉換時，對預測的後繼區域提前加分，使其進入 hotlist |
| **可見範圍** | 僅到達該 device 的 request | 部署於互連時可見所有 host 對所有 device 的 request；部署於 MC domain 時可見範圍與 CHMU 相當 |
| **跨 device 感知** | 不可能——Device 0 的 CHMU 看不到 Device 1 的流量 | 取決於部署位置——互連部署時自然可見，MC domain 部署時需軟體彙整 |
| **衰減機制** | Epoch-based 模式在 epoch 結束時 snapshot/reset counter；always-on 模式持續計數至超過 threshold | 所有 region 的溫度分數持續性週期右移遞減，反映即時活躍度變化 |

### 習知技術分類

| 類別 | 代表來源 | 做法 | 限制 |
|------|---------|------|------|
| OS fault-based 掃描 | Linux AutoNUMA、TPP (ASPLOS '23) | 定期將部分頁面 PTE 標記為 not-present，藉存取觸發的 page fault 判斷溫度並決定搬遷 | 毫秒級輪詢，對 AI workload 反應太遲；消耗 CPU 資源 |

| 軟體配置 / fault 驅動 | numactl、CUDA Unified Memory | numactl：程式設計師預先指定放置位置（靜態）；CUDA UM：GPU MMU 偵測到 page 不在本端時觸發 fault 驅動遷移（被動反應式） | 靜態配置無法適應動態存取模式；fault 驅動需等到實際存取 miss 才觸發，無預測能力 |
| OS 軟體 page metadata | Intel US12,443,537 | 在 kernel memory 中維護 page hotness metadata，以 callback 與 region filtering 降低偵測開銷 | OS 軟體層級方案；不含空間或序列維度評估 |
| CXL 裝置端硬體監測 | CXL 3.1 CHMU | 各 CXL Type 3 memory device 內嵌 Hotness Monitoring Unit，透過 hotlist ring buffer 回報（支援 polling / interrupt） | 各 device 獨立監測；CXL 3.1 spec 未規定空間或序列維度評估（內部演算法為 implementation defined）；純計數，無預測能力 |
| 存取頻率預測 | Huawei US12,379,861 | 以 CNN+LSTM 模型根據區塊歷史存取頻率時間序列預測未來存取頻率 | Storage 語境；需要 training data 與 inference 計算；非硬體狀態表的序列規律學習 |
| 純軟體 tiering | Chrono (EuroSys '25) | 以 timer 精確測量 page access frequency，自動調參進行溫度分類 | 不在硬體層執行，反應速度受限於軟體路徑延遲 |
| 推理/訓練協同調度 | Sirius (USENIX ATC '25) | runtime 軟體層 selective eviction：profiling 訓練資料溫度，inference burst 來時將 cold pages 搬至 host memory | 純軟體方案，需 runtime instrumentation；不含序列規律預測 |
| 延遲感知 tiering | Colloid (SOSP '24) | 基於各 tier 的實測存取延遲（loaded latency）在 tiers 間平衡熱資料分布 | 不在硬體層執行；依賴處理器端的延遲測量（如 CHA） |
| GPU 端硬體存取計數 | NVIDIA GPU Access Counter (Volta+), US10,361,722 | GPU 內建 per-page 硬體存取計數器，超過閾值時通知 driver 進行 page migration | 純計數，無預測能力；僅具 single-endpoint 可見性（只看到 GPU 端存取）；不含空間或序列維度評估 |
| HBM cache mode | Intel KNL MCDRAM, 各家 HBM cache 實作 | 硬體將 HBM 作為 DDR 的 last-level cache，以 cacheline 粒度自動管理搬遷 | Tag overhead 佔用 HBM 容量；cacheline 粒度與 AI workload 的 tensor/expert 粒度不匹配；工作集略超 HBM 容量時發生 thrashing；無預測能力，僅依據 LRU 等被動替換策略 |
| Markov Prefetcher | Joseph & Grunwald (ISCA '97), US5,778,436A | 以 prediction table 記錄 cache miss address 序列，predecessor/successor 結構，在 miss 發生時預測下一個可能 miss 的 cacheline | 學習的是 **address miss 序列**而非溫度狀態轉換序列；每次 cache miss 都觸發學習（高雜訊）；設計目標為 cache prefetch 而非跨 tier migration；缺乏聚合與信心度確認機制，不適用於高成本搬遷決策 |
| ML-based 預測 | AMD US12,174,742 (LSTM), Huawei US12,379,861 (CNN+LSTM), RecMG (LSTM for embeddings) | 以機器學習模型（LSTM、CNN+LSTM）從歷史存取模式預測未來存取頻率或 page 溫度 | 需要 training data；需要 inference 計算資源；需要 per-model / per-workload 配置調校。與本案的 lightweight table-based 方法屬於根本不同的技術路線 |
| RL-based adaptive tiering | ArtMem (ISCA '25) | 以 reinforcement learning agent 動態調整 migration threshold，最佳化搬遷策略 | 學習的是最佳 **policy parameters**（promotion/demotion threshold），而非 region-to-region 的溫度轉換序列；需要 training 過程 |
| Cache replacement | ARC (FAST '03)、LIRS (SIGMETRICS '02) | Cache 內部的替換決策演算法 | 僅處理 cacheline 粒度的 cache 內部管理，不涉及跨記憶體層級搬遷 |

### 新穎性

1. **溫度狀態轉換序列的自主學習與預測**：行為學習表自主觀察溫度轉換事件的先後順序（前驅/後繼/信心度），重複出現的對應關係累積信心度成為已確認的規律，長期未再出現則信心度衰減至自動失效。當新的溫度轉換事件發生時，評估引擎以該區域為前驅查詢行為學習表，對信心度超過確認閾值的後繼區域提前加分——實現對**尚未被存取**區域的溫度預測。此為所有已知前案均不具備的能力：CHMU、NeoMem、GPU Access Counter 僅做事後計數；AutoNUMA、TPP 依賴 fault 觸發的被動掃描；Markov Prefetcher 學習的是 address miss 序列而非溫度狀態轉換序列，且缺乏聚合過濾機制；ML-based 方案（AMD LSTM、Huawei CNN+LSTM、RecMG）需要 training data 與 inference 計算；ArtMem 學習的是 policy parameters 而非轉換序列。本案是唯一能在資料尚未被存取前預測其溫度轉換方向的方案。

2. **多維度評估的統一溫度分數架構**：時間維度（衰減式累計反映即時活躍度）、空間維度（鄰近區域擴散偵測）、序列維度（轉換序列預測）三者的加分結果共同寫入區域狀態表中的同一個溫度分數欄位，形成單一的複合溫度評估。現有方案中，無論是 CHMU 的單維度計數、NeoMem 的多指標分離回報、還是 ML-based 方案的模型輸出，均未將多個評估維度整合於同一硬體分數架構中。

3. **部署位置的彈性**：資料溫度評估引擎的演算法不依賴於特定部署位置。可部署於晶片內互連的流量匯聚點（觀察跨 MC domain 的全域流量）、MC domain 內部（觀察範圍與 CHMU 相當）、CXL switch（觀察跨 device 流量）、或以軟體/韌體形式實現。部署位置影響可見範圍但不改變核心演算法。此彈性使同一技術方案可適用於不同系統架構。

### 進步性

1. **預測式搬遷**：現有方案只能在資料已被存取後回報其溫度狀態；本案可在資料尚未被存取前預測其溫度轉換方向。搬遷因此可在 critical path 之外提前進行，避免存取時才發現資料不在近端記憶體的延遲代價。

2. **Lightweight 通用方案**：不需要 training data（vs ML 路線的 AMD LSTM、Huawei CNN+LSTM、RecMG）、不需要 per-model 配置（vs Pre-gated MoE、FineMoE 等 model-specific 方案）、不需要離線 profiling（vs MoE-Infinity、FAE 等需要事先 profiling 的方案）。行為學習表以固定大小的硬體 table 在運行期間自主學習，適用於任何具有可學習溫度轉換規律的 workload。

3. **與現有機制互補**：評估引擎只產生放置建議（placement hint），不替代現有記憶體管理機制。軟體決定是否及如何執行搬遷，可與 OS page migration、CUDA Unified Memory、CXL memory pooling 等既有機制共存。

### 103 防禦論述

**最可能的審查意見組合**：審查委員可能結合 Markov Prefetcher（predecessor/successor table 結構）與任何記憶體 tiering 前案（如 CHMU、AutoNUMA、TPP），主張本案為已知技術的顯而易見組合。

**防禦要點**：

第一，**聚合與過濾機制**。Markov Prefetcher 在每次 cache miss 時觸發學習，因為 cache miss 本身即為需要回應的事件（prefetch 的目的是減少 miss penalty）。本案的行為學習表僅在溫度轉換事件發生時觸發學習——而溫度轉換事件是經過時間維度的衰減式累計與閾值過濾後才產生的聚合結果。此聚合機制是將 Markov 結構從 cache prefetch 遷移至 tier migration 領域時必需的非顯而易見改進：cache prefetch 的成本低（一次 cacheline fetch），容許高 false positive rate；tier migration 的成本高（數十 KB 至數百 MB 的資料搬遷），不容許頻繁的錯誤預測。

第二，**學習對象的本質差異**。Markov Prefetcher 學習的是 address miss 序列——具體的記憶體位址之間的先後關係。本案學習的是溫度狀態轉換序列——region 從低溫度到高溫度的狀態變化之間的先後關係。前者是位址空間中的序列規律，後者是溫度狀態空間中的序列規律。此差異並非簡單的「把 address 換成 region」，而是觀察層級的根本轉換：從每次 miss 事件上升到經過聚合確認的狀態轉換事件。

第三，**成本不對稱性要求信心度確認**。本案的行為學習表包含信心度計數器與確認閾值機制——對應關係必須被重複觀測到足夠次數才會用於預測。Markov Prefetcher 不具備此機制，因為 prefetch 的錯誤成本僅為一次無用的 cache fetch。Tier migration 的錯誤成本（不必要的大規模資料搬遷）要求更高的預測確信度，此信心度確認機制是因應成本不對稱性而生的技術貢獻。

**參考先例**：AMD US12,174,742 以 method patent 形式保護 LSTM-based page placement prediction 的特定演算法，通過 35 U.S.C. §101 審查。本案同樣以特定演算法（table-based 溫度狀態轉換序列學習）應用於具體技術領域（記憶體層級放置），屬相同類型的可專利主題。

### 適用範圍與限制

**適用條件**：

- 運行期間存取模式以毫秒至秒級動態變化，且變化並非完全隨機——存在可從歷史溫度轉換事件序列中學習的統計規律
- 資料總量超過近端記憶體容量，需要跨層級放置決策
- 存取模式變化速度快於 OS 掃描週期（否則 AutoNUMA/TPP 等 OS 方案已足夠）

**不適用情境**：

- 靜態或 compile-time 即可確定的存取序列（如固定順序的 layer-by-layer inference，可直接以靜態排程處理）
- 線性增長模式（如 sequential scan，不產生溫度轉換事件）
- 變化週期在秒級以上的緩慢模式（OS 軟體方案已足以應對）

**行為學習表預測品質的影響因素**：

- **轉換序列的重複性**：重複性越高，信心度累積越快，預測準確率越高。典型有利場景為 MoE 模型中 expert 的啟用序列——同一 routing pattern 在多次 inference 中重複出現
- **Batch serving 交錯程度**：多個 request 的 expert 啟用序列交錯時，行為學習表觀察到的序列可能混雜多個 request 的 pattern，增加雜訊。信心度過濾機制可部分緩解此問題——真正重複的規律仍會累積信心度，而雜訊產生的偶發對應關係則因信心度不足而不被採用

## 五、專利構想摘要

### 核心概念

> 在具有複數個記憶體層級的資料處理系統中（flat mode），本案提出一種資料溫度感知方法。此方法的核心為一**資料溫度評估引擎（data temperature evaluation engine）**，以多維度存取行為評估產生各記憶體區域之**溫度分數（temperature score）**——時間維度追蹤存取強度並以可配置週期執行右移衰減；空間維度在區域發生溫度轉換事件時對鄰近區域加分（強度可配置，包括停用）；序列維度在區域發生溫度轉換事件時查詢**行為學習表（behavior learning table）**，對已學習之後繼區域加分。三個維度更新同一溫度分數欄位，形成複合評估。行為學習表於每次溫度轉換事件時自主記錄前驅/後繼對應關係，經重複觀測累積信心度並確認為序列規律。溫度分數跨越 promote 或 demote 閾值時，評估引擎將 hint entry 寫入 hotlist（ring buffer），以 interrupt 或 polling 方式通知軟體。軟體決定是否及如何執行搬遷。此方法以軟體或韌體實作為主，運行於系統處理器或專用微控制器，存取資訊透過硬體效能計數器或記憶體控制器統計取得。在需要更低延遲的場景中，亦可以硬體邏輯實作。

### 鎖定的核心特徵

| 核心特徵 | 說明 | 選擇理由 |
|---------|------|---------|
| **A. 行為學習表的溫度轉換序列自主學習** | 自主觀察溫度轉換事件的先後序列，重複出現的前驅/後繼對應關係累積信心度並確認為規律，長期未再出現則自動失效。能對**尚未被存取**之區域產生放置建議 | 所有已知 prior art（CHMU、NeoMem、GPU access counter、OS 掃描、Markov prefetcher、ML 預測）均僅能回報已被存取之區域，不具備溫度狀態轉換序列的預測能力。此為本案唯一無法被現有方案涵蓋的差異化 |
| **B. 多維度評估的統一溫度分數架構** | 時間、空間、序列三個維度更新同一張區域狀態表之同一溫度分數欄位，形成單一複合評估 | 統一的評估架構；現有方案最多僅涵蓋時間維度（存取計數），無空間擴散或序列預測 |
| **C. 部署彈性** | 以旁路方式取得存取資訊，不攔截亦不修改記憶體存取請求。可部署於互連匯聚點（集中式拓撲）、記憶體控制器域前端（分散式拓撲）、CXL switch、或以韌體/軟體實作 | 不限於特定互連架構或實作方式；核心新穎性在演算法本身，非部署位置 |

### 架構特色

- **自主評估與學習**：從取得存取資訊到評估溫度到學習序列規律到產生預測，全由評估引擎自主完成。軟體只需配置參數與回應放置建議
- **評估參數可配置**：衰減週期、promote/demote 閾值、各維度加分強度、信心度確認閾值等均可配置，並可依記憶體位址區間分別設定
- **感知與搬遷解耦**：評估引擎只產生放置建議，不直接操作記憶體。搬遷的決策與執行由軟體負責
- **對記憶體存取路徑零影響**：以旁路方式取得資訊，不在存取路徑上引入額外延遲
- **雙閾值遲滯判定**：promote 閾值與 demote 閾值分離，中間區維持現狀，避免衰減造成分數在閾值附近反覆震盪
- **區域粒度可配置**：區域大小由軟體配置，評估引擎以有限容量的狀態表追蹤，超出容量時以替換策略管理
- **適用場景明確**：存取模式在運行期間動態變化、具有可學習的統計規律、pattern 變化速度快於 OS 掃描週期的場景（MoE expert routing、推薦系統 embedding table 等）

## 六、實施例及應用例

### 實施例 1：LLM 推理服務 + LoRA 微調共用加速器
*LLM Serving + LoRA Fine-tuning Co-location*

**場景**：當前 data center 最常見的 co-location pattern——LLaMA 推理服務（chatbot，延遲敏感）與 LoRA 微調（背景任務）跑在同一張加速器上。架構上，上排 PE 跑推理，下排 PE 跑微調，兩者透過共享 L2 Cache 競爭同一張 HBM。

```
                ┌─────────────── AI Accelerator Chip ──────────────┐
                │                                                  │
                │  ┌── PE row 0 ──┐  ← LLaMA Inference (chatbot)   │
                │  │ PE PE PE..PE │     • Model weights            │
                │  └──────┬───────┘     • KV Cache (per request)   │
                │         │                                        │
  ┌──────┐      │  ┌──────┴───────────────┐                        │  ┌─────┐
  │ Host │ PCIe │  │   L2 Cache (Shared)  │              Memory I/F│  │     │
  │ CPU  │◄────►│  └──────┬───────────────┘                ◄──────►│  │ HBM │
  └──┬───┘NVLink│         │                                        │  │     │
     │          │  ┌──────┴───────┐  ← LoRA Fine-tuning (背景)     │  └─────┘
  ┌──┴───┐      │  │ PE PE PE..PE │     • Base model weights       │
  │ Host │      │  └──────────────┘     • Gradient / Optimizer st. │
  │ DRAM │      │                       • Batch data (epoch 切換)  │
  └──────┘      │                                                  │
                │  ┌── Processing Element (zoom) ─────────┐         │
                │  │ L1 Cache / Scratchpad ─► Tensor Core │         │
                │  │                       ─► Vector Unit │         │
                │  │                       ─► SFU         │         │
                │  └──────────────────────────────────────┘         │
                └──────────────────────────────────────────────────┘
```

**問題**：LoRA 微調每個 step 大量讀寫 gradient / optimizer state，存取頻率遠高於推理服務的間歇性 request。LRU 只看頻率，會把 LoRA 的資料判為「最熱」，把 LLaMA 的 KV Cache 與 weights 擠出 HBM——推理延遲飆升。

```
  ┌─────────── 沒有本案：LRU 判斷錯誤 ───────────┐
  │ HBM 內容：                                    │
  │   ▓▓ LoRA gradient        ← 高頻寫入，最熱   │
  │   ▓▓ LoRA weights         ← 高頻讀取         │
  │   ▓▓ LoRA batch data      ← 高頻讀取         │
  │   ░░ LLaMA KV Cache       ← 被擠出 ✗         │
  │   ░░ LLaMA weights        ← 被擠出 ✗         │
  │                                              │
  │ → 推理延遲飆升                                │
  └──────────────────────────────────────────────┘

  ┌─────────── 有本案：依區間配置 ───────────────┐
  │ 感知單元 CSR：                                │
  │   推理位址區間：長衰減週期 + 低 promote 閾值  │
  │     → 持續存取即可維持分數                    │
  │   訓練位址區間：短衰減週期 + 高 promote 閾值  │
  │     → epoch 切換後快速歸零 → demote          │
  │                                              │
  │ HBM 內容：                                    │
  │   ░░ LLaMA KV Cache       ← 穩定留在 HBM ✓   │
  │   ░░ LLaMA weights        ← 穩定留在 HBM ✓   │
  │   ▓▓ LoRA 當前 epoch 資料 ← 動態調整          │
  │   ░░ (LoRA 上個 epoch 已自動 demote)          │
  │                                              │
  │ → 推理延遲不受訓練影響                        │
  └──────────────────────────────────────────────┘
```

**本案配置**：軟體透過 CSR 為推理與訓練的位址區間分別配置感知參數，硬體依區間自動套用。

**效果**：

- LLaMA 的 KV Cache 與 weights 穩定留在 HBM，推理延遲不受訓練影響
- LoRA 上個 epoch 的資料隨 epoch 切換自動被 demote，騰出空間給新 batch
- 兩個工作負載動態共用 HBM，無需軟體手動切割空間，亦無需 runtime instrumentation

### 實施例 2：MoE 推理
*Mixture-of-Experts Inference*

```
                       ┌────────┐
                       │ Output │
                       └────▲───┘
                            ⊕  ◄── Gating 權重加權求和
                ┌───────────┼───────────┬────────┐
            ┌───┴───┐   ┌───┴───┐   ┌───┴───┐  ┌─┴────┐
            │Expert1│   │Expert2│   │Expert3│..│ExpertN│
            │ ★啟用 │   │ 未啟用 │   │ ★啟用 │  │ 未啟用│
            └───▲───┘   └───────┘   └───▲───┘  └──────┘
                └───────────┬───────────┘
                       ┌────┴─────┐
                       │  Gating  │
                       │ Network  │
                       └────▲─────┘
                       ┌────┴─────┐
                       │ Input    │
                       └──────────┘
```

**場景**：MoE 模型每層數百個 expert，HBM 僅能容納其中一部分。每個 token 由 Gating Network 選中 2~4 個 expert（圖中粉色），其餘不參與此 token 計算。

**問題**：

1. **反應式延遲**：被選中的 expert 若不在 HBM，須先搬入才能計算
2. **無跨 token 記憶**：Gating Network 只看當前 token，無法利用「Expert 1 → Expert 3」這類序列規律

**本案運作**：

- 時間策略對被存取的 expert 加分 → 跨越 promote 閾值 → 冷熱判定單元觸發冷轉熱事件
- 行為學習表觀察「Expert 1 變熱 → Expert 3 變熱」的順序，重複出現後確認為規律
- 下次 Expert 1 又冷轉熱時，預測性策略立刻對 Expert 3 加分 → promote hint
- 軟體在 Expert 3 真正被 Gating Network 選中**之前**就背景搬入 HBM
- 長期未啟用的 expert 因週期性右移歸零 → demote hint

**效果**：

- 重複出現的 expert 啟用序列從**反應式搬遷**變為**主動預搬遷**
- Expert 在實際被選中時已在 HBM，運算單元零等待開始計算
- 適用任何 routing 行為有統計規律的 MoE 模型，無需為每種模型客製 prefetch 邏輯

### 實施例 3：Chiplet 多晶粒架構
*Multi-Chiplet SoC Architecture*

```
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ Chiplet 0│  │ Chiplet 1│  │ Chiplet 2│
  └────┬─────┘  └────┬─────┘  └────┬─────┘
       │             │             │
  ═════╪═════════════╪═════════════╪═══════
  ║          晶粒間互連                  ║
  ║   ┏━━━━━━━━━━━━━━━━━━━━━━━━━┓     ║
  ║   ┃      感知單元             ┃     ║
  ║   ┃  Region X 聚合存取偵測   ┃     ║
  ║   ┃  + 主要來源識別           ┃     ║
  ║   ┗━━━━━━━━━━━━━━━━━━━━━━━━━┛     ║
  ═════╤═════════════╤═════════════╤═══════
       │             │             │
  ┌────┴────┐  ┌────┴────┐  ┌────┴────┐
  │近端     │  │近端     │  │近端     │
  │記憶體 0 │  │記憶體 1 │  │記憶體 2 │
  └─────────┘  └─────────┘  └─────────┘
```

**場景**：SoC 採用 chiplet 架構，每個 chiplet 各自連接近端記憶體，同時透過互連存取其他 chiplet 的記憶體（remote access）。

**問題**：某塊資料同時被多個 chiplet 存取形成跨晶粒熱點，但各 chiplet 的記憶體控制器只看到自己端的流量，無法察覺。

**本案運作**：

- 感知單元設置於晶粒間互連，觀察所有 chiplet 對所有記憶體的存取請求
- 時間策略累加聚合存取活動 → 分數上升
- 空間策略偵測鄰近 region 也有跨晶粒存取 → 加分修正
- 在互連協定提供來源識別資訊之場景下，hint 可附帶主要存取來源（如「主要被 Chiplet 1 存取」）
- 跨晶粒存取消失時，分數經週期性右移下降 → 觸發 demote hint

**效果**：

- 跨晶粒熱點被自動偵測，無需 OS 週期性 NUMA 掃描
- 軟體據主要存取來源決定搬遷目的地（搬至最頻繁存取的 chiplet 的近端記憶體）
- 存取模式改變後自動反應，無需等待下次 OS 掃描週期

## 八、Claim 規劃

### 裝置項

**1.** 一種資料溫度感知裝置，設置於一資料處理系統之晶片內互連中，該裝置包含：

一請求窺探電路，以旁路方式擷取通過該晶片內互連之記憶體存取請求的資訊；

一區域狀態表，為各記憶體區域維護一冷熱分數；

至少一存取行為追蹤電路，根據該請求窺探電路所擷取之資訊，以至少一種策略追蹤各記憶體區域之存取行為並更新該冷熱分數；

一判定電路，根據該冷熱分數與至少一可配置閾值之比較，產生放置建議；

其中，該請求窺探電路不攔截亦不修改該記憶體存取請求。

**2.** 如請求項 1 所述之裝置，其中該至少一存取行為追蹤電路包含一時間策略追蹤電路，持續追蹤各記憶體區域之存取活躍程度，並以一可配置週期對該區域狀態表中所有區域之冷熱分數執行右移操作，使未被持續存取之區域的冷熱分數逐步歸零。

**3.** 如請求項 1 所述之裝置，其中該至少一存取行為追蹤電路包含一空間策略電路，當一記憶體區域之冷熱分數從低於一閾值上升至高於該閾值時，對位址空間中相鄰之記憶體區域的冷熱分數加入正向修正。

**4.** 如請求項 1 所述之裝置，進一步包含一轉換序列規律預測電路，記錄冷熱轉換事件之前後區域對應關係，當該對應關係經重複觀測確認後，於偵測到已知前兆時對預測之後繼區域的冷熱分數加入修正。

**5.** 如請求項 1 所述之裝置，其中該至少一可配置閾值、右移衰減週期與修正量可由軟體透過組態暫存器設定，且配置可為全域預設或依位址區間分別設定。

**6.** 如請求項 1 所述之裝置，其中該晶片內互連為晶片內網路、交叉開關、或晶粒間互連之一。

**7.** 如請求項 1 所述之裝置，其中該資料處理系統包含一近端記憶體層級與一遠端記憶體層級，該放置建議指示各記憶體區域應被提升至該近端記憶體層級或降級至該遠端記憶體層級。

### 方法項

**8.** 一種資料溫度感知方法，於一資料處理系統之晶片內互連中執行，包含步驟：

以旁路方式擷取通過該晶片內互連之記憶體存取請求的資訊；

根據所擷取之資訊，以至少一種策略追蹤各記憶體區域之存取行為並更新一冷熱分數；

根據該冷熱分數與至少一可配置閾值之比較，產生放置建議；

其中，擷取步驟不攔截亦不修改該記憶體存取請求。

**9.** 如請求項 8 所述之方法，其中該至少一種策略包含一時間策略、一空間策略、及一預測性策略，三者更新同一冷熱分數。

**10.** 如請求項 9 所述之方法，其中該預測性策略包含記錄冷熱轉換事件之前後區域對應關係，經重複觀測確認後，於偵測到已知前兆時對預測之後繼區域的冷熱分數加入修正。

### 系統項

**11.** 一種資料處理系統，包含：

複數個處理元件；

一晶片內互連，連接該複數個處理元件；

一近端記憶體層級與一遠端記憶體層級；

一資料溫度感知裝置，設置於該晶片內互連中，包含一請求窺探電路、一區域狀態表、至少一存取行為追蹤電路、及一判定電路，該判定電路根據冷熱分數產生放置建議；

其中，軟體根據該放置建議決定是否執行該近端記憶體層級與該遠端記憶體層級之間的資料搬遷。

### 策略說明

| 決策 | 理由 |
|------|------|
| 獨立項使用「至少一種策略」 | 最小實作只需時間策略即可運作，保護範圍最寬 |
| 三種策略整合放在依附項（Claim 9） | 完整三策略組合作為 fallback 防線 |
| 預測性策略的序列規律學習放在依附項（Claim 4, 10） | 核心新穎性，必要時可併入獨立項強化 103 防禦 |
| 空間策略獨立為 Claim 3 | 與預測性策略分開保護，各自獨立 |
| CSR 可配置含位址區間粒度（Claim 5） | 涵蓋實施例 1 的多工作負載場景 |
| 互連類型不限定在獨立項 | Claim 6 依附項擴展至 NoC / crossbar / die-to-die |

## 九、§101 抽象概念風險評估（美國申請）

### 風險分析

在美國申請時，USPTO 審查委員可能依 35 U.S.C. §101（Alice/Mayo 二步測試）主張本案的核心概念——「追蹤存取行為並評估冷熱以產生放置建議」——屬於 abstract idea（如 mental process 或 mathematical concept）。

**Step 1（Abstract Idea 判定）**：Claim 1 獨立項描述的「窺探、追蹤、評估、產生建議」流程，若僅以功能性語言撰寫，可能被認定為可由人腦執行的 mental process。

**Step 2A Prong 2 + Step 2B（"Something More" 判定）**：需證明 claim 整體包含足以將抽象概念轉化為具體技術改進的額外要素。

### 防禦策略

| Claim | §101 防禦角色 | 關鍵要素 |
|:---:|---|---|
| **1** | 獨立項含「請求窺探電路」「區域狀態表」「存取行為追蹤電路」「判定電路」等具體硬體元件，且限定於「晶片內互連中」的物理位置 | 硬體裝置 + 物理位置限定 |
| **3** | **核心防禦項**：空間策略電路在冷轉熱事件發生時對鄰近區域加入分數修正，是即時回應硬體事件的具體電路行為，人腦無法在互連頻寬下即時執行 | 硬體事件驅動的即時修正 |
| **4** | **核心防禦項**：轉換序列規律預測電路，記錄前後區域對應關係並經重複觀測確認——具體的硬體資料結構與狀態機行為 | 硬體 pattern table + 確認機制 |
| **5** | 可配置閾值與右移衰減週期，支援依位址區間分別設定——具體的硬體暫存器配置機制 | 硬體 CSR 配置粒度 |
| **8** | 方法項通常比裝置項更易被認定為 abstract idea；防禦重點在於步驟限定「於晶片內互連中執行」，強調方法整體產生具體的技術效果（以硬體速度主動感知冷熱並產生放置建議，改善記憶體層級間的資料放置效率） | 硬體執行限定 + 技術效果 |

**建議**：若遭 §101 rejection，以 Claim 4（轉換序列規律預測電路）為核心反駁錨點，強調其硬體資料結構與狀態機行為是人腦或通用軟體無法即時執行的。必要時將 Claim 4 的限定併入獨立項以強化抗辯。Claim 3（空間策略電路）可作為備用錨點。

## 十、侵權判定與推廣

### 侵權判定方法

| Claim | 偵測方法 | 證據來源 |
|:---:|---------|---------|
| **1** | 確認系統是否存在設置於晶片內互連中的存取監測機制，且該機制以旁路方式運作、不影響資料路徑延遲 | 硬體技術文件、CSR register map |
| **3** | 觀察記憶體搬遷是否呈現空間擴散行為：某個 region 被 promote 後，相鄰 region 是否在短時間內也被 promote | 效能測試 |
| **4** | 觀察搬遷是否呈現序列預測行為：資料是否在被實際存取之前就被搬入近端記憶體（如 Layer N 搬遷後 Layer N+1 立即被搬入，早於實際存取） | 效能測試 |
| **5** | 確認是否存在可配置的冷熱評估參數（右移衰減週期、閾值），且支援依位址區間分別設定 | SDK 文件、driver source code |
| **7** | 確認系統是否具備近端與遠端記憶體層級，且搬遷方向與感知單元的 hint 一致 | 系統架構文件 |

**侵權判定建議路徑**：

先透過公開文件確認 Claim 5（CSR 中是否有冷熱評估相關的可配置參數），建立「該硬體具備冷熱感知機制」的事實基礎；再以效能測試觀察 Claim 4（搬遷是否呈現預測行為），佐證感知機制包含預測性策略。

### 推廣應用

| 場景 | 說明 |
|------|------|
| CXL memory tiering | 在 CXL switch 或 host 端互連中設置感知單元，管理 local DRAM 與 CXL-attached memory 間的資料放置 |
| GPU 異質記憶體 | GPU 的 HBM 與 host memory 間的資料放置管理 |
| Chiplet 多晶粒架構 | 跨 chiplet 的記憶體資料放置優化（如實施例 3） |
| 通用 NUMA 系統 | 在 NUMA 互連中設置感知單元，優化跨 node 的資料放置 |
